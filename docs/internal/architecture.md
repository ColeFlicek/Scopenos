# Scopenos Architecture

Internal reference. Not for public docs — contains rationale for rejected designs and known tradeoffs.

---

## The four-layer model

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 4: Decision Memory                                   │
│  Why things are the way they are. Survives code changes.    │
│  Tools: log_decision, get_decision_history, query_decisions │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: Invariant Contracts                               │
│  Rules the codebase must not violate. Enforced post-commit. │
│  Tools: create_contract, check_contracts, approve_contract  │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: Semantic Embeddings                               │
│  What functions do, queryable by meaning not name.          │
│  Tools: query_similar_functions, performance detectors      │
├─────────────────────────────────────────────────────────────┤
│  Layer 1: Call Graph                                        │
│  Who calls whom, what files exist, what signatures look like│
│  Tools: get_callers, get_callees, get_impact_radius, ...    │
└─────────────────────────────────────────────────────────────┘
```

Each layer uses the one below. Contracts reference functions from the call graph. The performance detector uses embeddings to classify functions semantically and the call graph to trace N+1 paths.

---

## Layer 1: Call Graph

**What it stores:** `function_nodes` (id, name, file, module, signature, body, start_line, end_line, is_async, return_type, parameter_names, enclosing_class, structural_layer, summary) and `call_edges` (caller_id → callee_id, is_external).

**Parser pipeline:**
1. Language-specific precision parsers (Python via tree-sitter, JS/TS via tree-sitter, Swift, Kotlin, PHP)
2. Generic fallback parsers for 12+ languages (Zig, Groovy, Perl, etc.)
3. SCIP ingestion for type-resolved edges from compiled languages

**`structural_layer` field:** marks whether a function's context came from precision parsing (`"precision"`) or generic fallback (`"generic"`). Used to calibrate confidence in analysis results.

**Incremental indexing:** `index_changes` accepts a dict of `{filepath: content}` and updates only those functions. The post-commit hook calls this on every commit. Full re-index via `index_project` is needed only after adding a new parser or changing the schema.

**Why Postgres not SQLite:** SQLite single-writer bottleneck becomes real with concurrent project indexing. Postgres + pgvector was chosen before public launch rather than after. See `log_decision` records tagged `trigger: "git:03df6fd"`.

## Layer 2: Semantic Embeddings

**What it stores:** `function_embeddings` (project_id, function_id, embedding vector(1536)) with HNSW index for cosine similarity search.

**Embedding model:** `text-embedding-3-small` (OpenAI) by default. The `EMBEDDING_PROVIDER`, `EMBEDDING_MODEL`, `EMBEDDING_DIM` env vars configure this. Ollama is supported for local/air-gapped deployments.

**Two embedding types:**
- Function embeddings: indexed at `index_project` / `index_changes` time
- Decision embeddings: indexed when `log_decision` is called (for `query_decisions`)
- Contract embeddings: indexed when `approve_contract` is called (for `check_contracts`)

**Enrichment:** Functions without docstrings get a body-text embedding, which is lower quality. `enrich_summaries` runs Claude Haiku on these to generate summaries before re-embedding. The `needs_enrichment` flag (TODO: see roadmap) should replace the current `embedding_model` string check.

**HNSW index:** Applied via `CREATE INDEX ... USING hnsw (embedding vector_cosine_ops)`. Not applied in the current K8s deployment (see roadmap known issue). Fine at current scale.

## Layer 3: Invariant Contracts

**What it stores:** `contracts` table (id, title, natural_language, structural_expression JSON, violation_examples[], compliance_examples[], status, project_ids).

**Lifecycle:** draft → approved → (active enforcement) → deactivated

**Enforcement:** The post-commit hook calls `POST /api/contracts/check` with new function IDs after every commit. Violations are stored in `contract_violations` and surfaced in the web UI.

**Primary adversarial vector:** An agent that can delete a contract can bypass enforcement. This is why DELETE /api/contracts was removed — deactivation (POST .../deactivate) is the only exit path, and it preserves the audit trail.

**`structural_expression`:** JSON blob encoding the abstract rule as a set of detectable patterns (e.g. "all public functions must have a docstring", "no HTTP calls inside a loop"). Generated by Claude Haiku at contract creation time. The structural expression drives embedding-based violation scoring.

---

## Layer 4: Decision Memory

**What it stores:** `decisions` table (id, project_id, type, description, rejected_alternatives, trigger, linked_function_ids[], created_at) + decision embeddings.

**Why it exists:** Code is the answer; decision memory is the question behind the answer. Without it, every agent starts from scratch and may re-implement rejected approaches.

**Current gap:** `risk_detection_mode: "structural_heuristic_no_decisions"` in every project means no decisions are linked to functions. The backfill script (`scripts/backfill_decisions.py`) was never run, and the post-commit hook logs commit-level decisions but doesn't always populate `linked_function_ids`. This degrades `get_decision_history` quality. Fix: run the backfill script against all indexed projects.

## Entry points and routing

**MCP server:** FastMCP over HTTP/SSE. All `@mcp.tool()` decorated functions are MCP tools. Auth middleware (`AuthMiddleware`) fires on every MCP message, sets `_current_user` ContextVar via `X-API-Key` header lookup.

**HTTP routes:** `@mcp.custom_route(...)` decorators register Starlette routes. Read routes are unauthenticated. Write routes call `_require_valid_key(request, db)`. Two routes are intentionally unauthenticated: `POST /index` and `POST /api/decisions` (used by git hooks without user credentials).

**Web UI:** Single-page app served from `GET /`. All JS is inlined in `src/web/template.py`. Write operations include `X-API-Key` header loaded from localStorage.

**Background workers:** Redis + RQ. `index_project` and `enrich_summaries` enqueue jobs; `GET /api/jobs/{id}` polls status. `index_changes` is synchronous (fast).

## Key architectural decisions

**ContextVar for per-request user (not threading.local):** FastMCP is async; thread-local state doesn't work across `await` boundaries. ContextVar propagates correctly through async task graphs. Services is a singleton — user is per-request.

**`_get_services()` pattern:** Returns a `Services` singleton (db, embeddings, indexer, contracts). Initialized once at startup via FastMCP lifespan. Avoids repeated DB connection setup and ensures all handlers share the same connection pool.

**Subsystem detection via module prefix clustering:** `get_project_home` groups functions by module prefix. No explicit subsystem configuration needed. Works well for Python (module paths reflect architecture) and reasonably for other languages.

**Demo projects are read-only at the storage layer:** `check_project_access` returns False for write operations on demo projects regardless of who's asking. Not enforced at the API key level — a user can own all demo repos and still can't write to them. This is intentional: demo data is shared across all users.

## Known non-obvious constraints

- `asyncpg` connection pool (min=2, max=10 currently, should be raised before high-concurrency load)
- The `app.src.*` namespace duplication in the call graph is a Docker build artifact — `app.src.module` and `src.module` may appear as separate nodes for the same function. Pending cleanup (see roadmap).
- `enrich_summaries` model check: `embedding_model = 'text-embedding-3-large'` string literal identifies unenriched functions. Should be a `needs_enrichment BOOLEAN` column.
- Post-commit hook requires `SCOPENOS_URL` env var to be set on the developer's machine. Without it, commits silently skip indexing.
