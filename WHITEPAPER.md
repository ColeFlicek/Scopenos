# ACIP: AI Code Intelligence Platform
## Architecture, Design, and Optimization Opportunities

**Version:** Post-multi-project migration  
**Date:** June 2026  
**Status:** Internal review draft

---

## 1. What ACIP Is

ACIP is a self-hosted code intelligence server. It indexes a codebase into three complementary knowledge stores, then exposes them via MCP (Model Context Protocol) to Claude Code sessions. The goal is to replace expensive, repetitive file reads with structured, semantically-aware queries — so that a language model asking "what calls this function?" or "why was this designed this way?" gets a precise, pre-computed answer instead of scanning source files.

ACIP is designed around one constraint: **no external databases, no external vector stores, no cloud dependencies beyond the embedding API**. Everything lives in a single SQLite file.

## 3. Three-Layer Architecture

### Layer 1 — Call Graph

**What it stores:** Every function, method, and class in the codebase as a node. Every call, inheritance, and import relationship as a directed edge.

**How it works:**

1. **Parsing** (`src/call_graph/parser.py`) — Uses `tree-sitter` to parse Python and TypeScript source files into ASTs. Two visitors (`_visit_python`, `_visit_typescript`) walk the AST and emit:
   - `FunctionNode`: id (module-qualified), name, file, type (function/method/class), signature (first line), docstring, body (truncated at 2000 chars), and `body_hash` (SHA-256[:16] of full function text).
   - `CallEdge`: caller_id, callee_name (unresolved), edge_type (calls/inherits/imports), file.

2. **ID scheme** — Node IDs are module-relative dotted paths, e.g. `src.call_graph.storage.CallGraphDB.upsert_nodes`. The `project_root` parameter drives `_path_to_module()` which strips the root prefix and converts slashes to dots.

3. **Storage** (`src/call_graph/storage.py`) — All nodes/edges go into SQLite. The `nodes` table has a composite primary key `(project_id, id)`. The `edges` table resolves callee names at write time: `_resolve_callee()` does exact match, then suffix match (`.func_name`), then stores the bare name as-is for unresolved externals.

4. **Queries** — `get_callers`, `get_callees`, `get_impact_radius` (BFS up to depth N). All are O(edges) SQL queries with indexed joins.

**Body hash:** Each function stores `sha256[:16]` of its full text. On re-index, only functions whose hash changed get re-embedded. This is the core of incremental indexing — unchanged functions skip the entire embedding pipeline.

---

### Layer 2 — Semantic Embeddings

**What it stores:** One dense vector per function, generated by an embedding model. Enables nearest-neighbor search: "find functions semantically similar to this snippet."

**How it works:**

1. **Chunking** (`src/embeddings/chunker.py`) — Each `FunctionNode` is converted to a `FunctionChunk` with a formatted embed text: `Function: {id}\nSignature: {sig}\nDocstring: {doc}\nSummary: {summary}\nBody:\n{body[:2000]}`.

2. **Summarization** — Before embedding, each function gets a one-sentence LLM-generated summary (Claude Haiku). This summary enriches the embed text with semantic intent that raw code may not convey. Summaries are cached in `nodes.summary` and reused on subsequent re-indexes.

3. **Embedding** (`src/embeddings/embedder.py`) — Calls the configured embedding provider (OpenAI `text-embedding-3-small` by default, or a local Ollama model). Batches of 100 functions per API call.

4. **Storage** — Vectors go into a `sqlite-vec` `vec0` virtual table (`function_embeddings`). The embedding ID format is `{project_id}::{node_id}` — this prefixing is the multi-project isolation mechanism, since `vec0` cannot have additional filter columns.

5. **Query** — `query_similar_functions` embeds the query snippet, runs KNN via `vec0`'s `WHERE embedding MATCH ? AND k = N`, then JOINs back to `nodes` via `substr(id, instr(id, '::') + 2)` to strip the project prefix. Optionally filtered by project.

**Incremental update flow:**
```
file changes → parse new nodes → compare body_hash vs. stored hash
→ delete_by_ids(changed | deleted) → upsert_nodes → upsert_edges
→ upsert_chunks(changed_ids only) → embed → store
```
Only changed functions touch the embedding API. A one-line fix to a 100-function file embeds one function, not 100.

---

### Layer 3 — Decision Memory

**What it stores:** Structured records of every design decision, linked to the functions they govern, with semantic search over their reasoning.

**How it works:**

1. **Schema** — Two SQLite tables: `decisions` (id, project_id, type, description, rejected_alternatives, trigger, created_at) and `decision_functions` (decision_id → function_id many-to-many join).

2. **Log path** — `log_decision()` takes a decision record → generates reasoning text (type + description + rejected alternatives + trigger + function names) → embeds it into `decision_embeddings` vec0 table → writes the structured record to SQLite → links function IDs.

3. **Get path** — `get_decision_history(function_name)` does a SQL lookup: `find_node_by_name` → JOIN `decision_functions` → JOIN `decisions`. Pure SQLite, no embeddings involved. Returns every decision ever linked to that function.

4. **Semantic query** — `query_decisions(text)` embeds the query → KNN over `decision_embeddings` → fetches matching decision records → returns with similarity scores. Finds prior decisions by intent, not by function name.

5. **Auto-population** — The post-commit git hook calls `/api/decisions` on every commit, linking the decision to the specific function IDs that changed (resolved via `/api/functions` endpoint). The backfill script does the same for historical commits.

## 4. Multi-Project Support

Each of the three layers is namespaced by `project_id`:

| Store | Isolation mechanism |
|---|---|
| `nodes` | `(project_id, id)` composite PK |
| `edges` | `project_id` column, indexed |
| `decisions` | `project_id` column |
| `function_embeddings` | ID prefix `{project_id}::` (vec0 limitation) |
| `decision_embeddings` | UUID IDs; project filter via JOIN with `decisions` |

**Project ID derivation (priority order):**
1. Explicit parameter in API call
2. `ACIP_PROJECT` env var (in hooks/scripts)
3. Git remote URL basename (e.g. `github.com/org/myapp.git` → `myapp`)
4. Directory basename of project root

**Schema migration:** At server startup, `CallGraphDB.init()` detects the old single-project schema (no `project_id` column on `nodes`) and runs an in-place migration: rebuilds `nodes` with composite PK, adds `project_id` to `edges` and `decisions`, and registers existing data under `project_id='default'`. Old embeddings (no `::` prefix) are detected and wiped by `EmbeddingStore.init()` on first post-migration startup.

## 5. Data Flow: Indexing a Project

```
/acip-import or index_project called
         │
         ▼
Collect .py/.ts/.tsx files
         │
         ▼
tree-sitter parse → FunctionNode[] + CallEdge[]
         │
         ▼
Snapshot old_hashes (node_id → body_hash) from DB
         │
         ▼
Compute changed_ids = {nodes where new_hash ≠ old_hash}
Compute deleted_ids = {nodes in DB but not in new parse}
         │
         ├─► delete_by_ids(changed ∪ deleted) from function_embeddings
         │
         ├─► delete_file_data(file) → wipe nodes + edges for that file
         │
         ├─► upsert_nodes(all new nodes, project_id)
         │
         ├─► upsert_edges(all new edges, project_id)
         │
         └─► [only for changed_ids]
              extract_chunks → LLM summary → embed → upsert_chunks
```

## 6. Data Flow: Query (MCP Tool Call)

```
Claude calls get_callers("upsert_chunks", project_id="acip")
         │
         ▼
find_node_by_name("upsert_chunks", project_id="acip")
  → exact id match, then name match, then suffix LIKE match
         │
         ▼
SELECT callers FROM edges JOIN nodes WHERE callee_id = ? AND project_id = ?
         │
         ▼
Return [{id, name, file, module, signature, edge_type, project_id}]
```

```
Claude calls query_similar_functions(snippet, top_k=5, project_id="acip")
         │
         ▼
embed(snippet) → 1536-dim vector
         │
         ▼
KNN query on function_embeddings (vec0):
  WHERE embedding MATCH <vector> AND k = 25  (5×5 for project filter headroom)
         │
         ▼
JOIN nodes ON id = substr(knn.id, instr(knn.id, '::') + 2)
           AND project_id = 'acip'
LIMIT 5
         │
         ▼
Return [{id, name, file, signature, summary, distance, project_id}]
```

## 7. Key Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Database | SQLite (single file) | Zero ops, portable, sufficient for <10K functions |
| Vector storage | sqlite-vec (vec0) | Same file as call graph, no Postgres/Chroma/Pinecone required |
| Parser | tree-sitter | Language-specific ASTs, no regex heuristics, 10-50ms per file |
| Embedding model | OpenAI text-embedding-3-small (default) | Good quality/cost; swappable to local Ollama |
| Summary model | Claude Haiku | Fast, cheap; enriches embed text with semantic intent |
| Incremental hash | SHA-256[:16] of full function text | 16 hex chars = 64-bit — collision probability negligible for codebases |
| Project isolation | Prefix on vec0 IDs | vec0 has no WHERE on non-vector columns in KNN subquery |
| Multi-project migration | In-place schema upgrade at startup | Zero-downtime, no separate migration CLI needed |

## 8. Optimization Opportunities

The following are areas where performance, cost, or quality can be significantly improved. They are ordered by estimated impact.

---

### 8.1 Embedding Batching Across Files (HIGH IMPACT — Cost & Speed)

**Current behavior:** `index_project` and `index_changes` collect all changed chunks across files, then call `upsert_chunks` once with all of them. `upsert_chunks` calls `_generate_summary` sequentially — one LLM call per function before any embedding happens.

**Problem:** Summarization is serial. For a 50-function initial index, that's 50 sequential LLM round-trips before a single embedding vector is created. If each summary call takes ~1s, that's 50s of sequential latency.

**Optimization:** Parallelize summarization with `asyncio.gather`. Since summaries are independent, all 50 can run concurrently:
```python
summaries = await asyncio.gather(*[_generate_summary(c) for c in needs_summary])
```
This would reduce summary time from O(n) sequential to O(1) parallel (bounded by API rate limits). With Haiku at ~100 RPM, a 50-function index would take ~30s instead of ~50s.

**Additional:** The embedding API already batches 100 texts per call, but summaries gate that. Parallelizing summaries unblocks the batch embedder sooner.

---

### 8.2 KNN Over-Fetch for Project-Filtered Queries (MEDIUM IMPACT — Query Quality)

**Current behavior:** When `query_similar` is called with a `project_id`, it fetches `top_k × 5` candidates from the KNN index (e.g. k=25 for top_k=5), then filters to the target project via JOIN. If that project has very few functions relative to total indexed functions, the returned list may still be fewer than `top_k`.

**Problem:** The 5× multiplier is a guess. A database with 10 projects of equal size needs 5× headroom; a database where one project has 90% of functions needs almost no headroom.

**Optimization:** Compute per-project fraction at query time:
```python
fraction = project_node_count / total_node_count
k_inner = max(top_k, int(top_k / fraction * 1.5))
```
This adapts the over-fetch to the actual project distribution, reducing wasted embedding comparisons.

**Alternative:** Add a `shadow_embedding_id` column to `nodes` that stores the prefixed embedding ID. Then filter in the KNN query using a `vec0` shadow table scoped to a project. This is a larger change but eliminates over-fetch entirely.

---

### 8.3 Body Hash Length (LOW IMPACT — Correctness)

**Current behavior:** `body_hash = sha256(func_text)[:16]` — 16 hex characters = 64 bits of collision resistance.

**Problem:** For a codebase with 10,000 functions, the birthday probability of a collision is ~2.7×10⁻¹⁵. Negligible. But if body_hash collides, a changed function would be considered unchanged and skip re-embedding, leaving a stale vector.

**Optimization:** Trivially extend to 32 chars (128 bits) — zero behavioral change, essentially zero collision risk at any realistic scale:
```python
body_hash = hashlib.sha256(func_text.encode()).hexdigest()[:32]
```

---

### 8.4 Edge Resolution: Unresolved Callees (MEDIUM IMPACT — Graph Quality)

**Current behavior:** `_resolve_callee` does:
1. Exact match against all known node IDs
2. Suffix match (`.func_name`) — finds `module.func_name` from bare `func_name`
3. Falls back to storing the bare name

**Problem:** External calls (`requests.get`, `json.dumps`, `asyncio.gather`) are stored as bare unresolved names. Calls to functions in other modules of the same project may fail suffix resolution if there are multiple matches (e.g. two classes both have `__init__`). These show up as phantom edges and inflate `get_callees` results.

**Optimization A:** Multi-match disambiguation — when suffix matching returns multiple candidates, prefer the one in the same module/package as the caller. This is a heuristic but correct ~90% of the time.

**Optimization B:** Track resolution confidence. Add a `resolved: bool` column to `edges`. Queries can optionally exclude unresolved edges. `get_impact_radius` on unresolved edges produces noisy BFS results.

**Optimization C:** Scope-aware resolution — the parser already tracks `parent_class` and `enclosing_func`. Pass this context to `_resolve_callee` to disambiguate `self.method()` calls by looking up the class's known methods.

---

### 8.5 sqlite-vec Dimension Mismatch Recovery (LOW IMPACT — Ops)

**Current behavior:** If the embedding model is changed (e.g. from `text-embedding-3-small` 1536d to `nomic-embed-code` 768d), the server raises `RuntimeError: Embedding dimension mismatch` on startup. Recovery requires manually deleting the SQLite file.

**Problem:** Data loss. All call graph data and decisions are in the same file as embeddings — deleting it resets everything.

**Optimization:** On dimension mismatch, drop and recreate only the `function_embeddings` and `decision_embeddings` vec0 tables, then update `_embedding_meta`. The call graph and decision records survive. Implement as `EmbeddingStore._handle_dim_mismatch()` triggered when the check fails.

---

### 8.6 Decision Linkage Quality (MEDIUM IMPACT — Decision Memory Usefulness)

**Current behavior:** The post-commit hook resolves changed files → function IDs via `/api/functions`. This correctly links decisions to specific functions. But the backfill script still uses file-path-to-module approximation (`src/foo/bar.py` → `src.foo.bar`) which produces module-level IDs, not function-level IDs.

**Problem:** Historical decisions linked to module-level IDs are not returned by `get_decision_history(function_name)` because that query matches function IDs, not module IDs.

**Optimization:** Update `backfill_decisions.py` to call `/api/functions` for each commit's changed files (same as the hook does), getting actual function IDs. This requires the ACIP server to be reachable during backfill, which it usually is. Add a `--use-api` flag (default on) vs `--use-module-paths` for offline mode.

---

### 8.7 Incremental Edges (LOW-MEDIUM IMPACT — Index Speed)

**Current behavior:** On every file change, the call graph for that entire file is wiped (`delete_file_data`) and re-parsed. This is correct because edges don't have per-function granularity — one function adding a call to another changes the edges without changing the caller's body hash.

**Problem:** For a 50-function file where one function changed, we re-parse and re-insert 50 nodes and all their edges. The parse is fast (10-50ms) but the DB writes are proportional to file size.

**Optimization:** Track edge hashes per function (a hash of the set of callees). Compare before deletion — only delete and re-insert edges for functions whose callee set changed. This reduces DB writes from O(file_size) to O(changed_functions).

**Complexity note:** This is more complex than body hashing because edges have a many-to-many relationship. Likely not worth it until the file size consistently exceeds ~200 functions.

---

### 8.8 `query_similar` Result Enrichment (LOW IMPACT — UX)

**Current behavior:** `query_similar_functions` returns `{id, name, file, module, signature, summary, distance, project_id}`. The `distance` is raw L2 distance from sqlite-vec.

**Problem:** L2 distance is not intuitive. The current `query_decisions` normalizes to a 0–1 similarity score, but `query_similar_functions` does not.

**Optimization:** Normalize distance to similarity in `query_similar`:
```python
# For unit-normalized embeddings, L2 distance ∈ [0, 2]
result["similarity"] = round(1.0 - distance / 2.0, 4)
```
Also: add `callers_count` (degree) to results — high-degree functions appearing in search results signal shared utilities worth knowing about.

---

### 8.9 Web UI Project Selector (LOW IMPACT — UX)

**Current behavior:** The web dashboard at `/ui` shows global stats (total nodes, total embeddings, total decisions). The projects list is shown but there's no way to scope queries to a project from the UI.

**Optimization:** Add a project selector dropdown to the dashboard. When a project is selected, the stats panel shows per-project counts, and a "search" panel lets you run `query_similar` and `query_decisions` scoped to that project. This doesn't change any backend code — it's a UI-only addition using the existing `/api/projects` and HTTP MCP endpoints.

---

### 8.10 `_ensure_indexes` on Every Startup (NEGLIGIBLE — Ops)

**Current behavior:** `_ensure_indexes()` runs on every startup, calling `DROP INDEX IF EXISTS idx_edges_unique` and recreating it. This is a write operation on a table that may have thousands of rows.

**Problem:** Unnecessary I/O on a path that should be a no-op after the first migration.

**Optimization:** Add a `_meta` table entry `indexes_v2_created: 1` that persists after migration. Check this flag in `_check_and_migrate` — if set, skip `_ensure_indexes` entirely.

## 10. What ACIP Is Not

- **Not a language server.** It does not provide completions, hover, or go-to-definition. It provides graph traversal and semantic search at a higher level of abstraction.
- **Not real-time.** The index is updated on commit (via hook) or manually. There is no filesystem watcher.
- **Not a code reviewer.** It surfaces structure and history; Claude Code does the reasoning.
- **Not multi-tenant.** All projects share one SQLite file and one server process. There is no auth, no isolation between projects beyond the `project_id` namespace, and no rate limiting.

## 11. Future Directions

1. **Cross-project similarity** — `query_similar_functions` already returns cross-project results when `project_id` is omitted. This could power "find where we solved this problem in another codebase" queries.

2. **Dependency graph between projects** — if two indexed projects import each other (monorepo), edges could cross project boundaries. The `_resolve_callee` suffix match already handles this if both projects are in the same DB.

3. **TypeScript type graph** — the current TypeScript parser extracts function calls and class inheritance, but not type relationships. Adding type edges would enable "what types does this function consume/produce?" queries.

4. **Decision aging** — decisions older than N days without a linked commit touching the governed function could be flagged as potentially stale. Add a `staleness_score` to `get_decision_history` results.

5. **Streaming re-index** — for large projects (>500 functions), emit progress events over SSE so the client can show indexing progress without polling.
